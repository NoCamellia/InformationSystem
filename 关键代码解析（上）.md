## 1. 统一返回结果封装 Result.java

这是整个后端API的基础，所有接口都通过这个类返回数据。

```java
@Data
public class Result<T> implements Serializable {
    private Integer code;    // 200成功，500失败
    private String message;  // 提示信息
    private T data;          // 泛型数据
    
    public static <T> Result<T> success(T data) {
        Result<T> result = new Result<>();
        result.setCode(200);
        result.setMessage("操作成功");
        result.setData(data);
        return result;
    }
    
    public static <T> Result<T> error(String message) {
        Result<T> result = new Result<>();
        result.setCode(500);
        result.setMessage(message);
        return result;
    }
}
```

**为什么这样设计**：泛型T可以返回任意类型的数据。前端只需判断code是否为200就知道请求成功与否。这是前后端分离项目的标准做法，如果后续需要添加请求ID、时间戳等字段，只需修改这一个类。

**可能的问题**：为什么不直接返回数据而要多加一层包装？

回答：直接返回数据时，前端无法区分请求成功还是失败。比如返回null，不知道是数据不存在还是服务器错误。统一封装后，前端始终能拿到code和message，处理逻辑更清晰。

## 2. 文章实体类 Article.java

```java
@Data
@TableName("article")  // 映射数据库表名
public class Article implements Serializable {
    
    @TableId(type = IdType.AUTO)  // 主键自动递增
    private Long id;
    
    private String title;
    private String summary;
    private String coverImage;
    private String content;       // 富文本HTML内容
    private Long categoryId;      // 外键，关联category表
    private Long authorId;        // 外键，关联admin表
    private String authorName;    // 冗余字段，避免join查询
    private Integer viewCount;    // 浏览量
    private Integer likeCount;    // 点赞数
    private Integer collectCount; // 收藏数
    private Integer isTop;        // 是否置顶 1/0
    private Integer isHot;        // 是否热门 1/0
    private Integer isRecommend;  // 是否推荐 1/0
    private Integer status;       // 0草稿 1发布 2下架
    private LocalDateTime publishTime;
    
    @TableField(fill = FieldFill.INSERT)
    private LocalDateTime createTime;
    
    @TableField(fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updateTime;
}
```

**为什么这样设计**：authorName是冗余字段，存储了作者名称。查询文章列表时不需要join admin表，提高了查询效率，以空间换时间。@TableField注解配合MyMetaObjectHandler实现时间字段自动填充。

**可能的问题**：为什么存authorName而不是通过authorId关联查询？

回答：数据库设计中的反范式化。在高并发场景下，减少join查询可以大幅提升性能。作者名称基本不会改变，冗余存储的代价很小，但能避免每次查询文章列表时都join admin表。

## 3. 自动填充时间 MyMetaObjectHandler.java

```java
@Component
public class MyMetaObjectHandler implements MetaObjectHandler {
    
    @Override
    public void insertFill(MetaObject metaObject) {
        // 插入数据时，自动填充创建时间和更新时间
        this.strictInsertFill(metaObject, "createTime", LocalDateTime.class, LocalDateTime.now());
        this.strictInsertFill(metaObject, "updateTime", LocalDateTime.class, LocalDateTime.now());
    }
    
    @Override
    public void updateFill(MetaObject metaObject) {
        // 更新数据时，自动填充更新时间
        this.strictUpdateFill(metaObject, "updateTime", LocalDateTime.class, LocalDateTime.now());
    }
}
```

**为什么这样设计**：实现MetaObjectHandler接口，在数据操作时自动填充时间字段。业务代码中不需要手动设置时间，减少重复代码。这是AOP思想的体现，在不修改业务代码的情况下统一处理了时间字段。

**可能的问题**：这和在数据库设置DEFAULT CURRENT_TIMESTAMP有什么区别？

回答：数据库默认值只在SQL层面生效，Java代码获取到的对象时间字段可能为null。MyBatis Plus的自动填充在Java层面处理，代码拿到的对象时间字段一定有值，更安全。

## 4. 跨域配置 CorsConfig.java

```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")            // 所有接口都允许跨域
                .allowedOriginPatterns("*")   // 允许所有来源
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                .allowedHeaders("*")           // 允许所有请求头
                .allowCredentials(true)        // 允许携带Cookie
                .maxAge(3600);                 // 预检请求缓存1小时
    }
}
```

**为什么这样设计**：前后端分离后，前端运行在localhost:3000，后端运行在localhost:8080，端口不同触发浏览器同源策略，导致跨域问题。CORS配置通过在响应头中添加允许字段来解决。

**可能的问题**：什么是跨域？为什么会有跨域问题？

回答：跨域是浏览器的安全限制，禁止网页脚本访问不同协议、域名或端口的资源。这个限制是为了防止恶意网站通过脚本访问用户在其他网站的数据。CORS是W3C标准，允许服务器声明哪些来源可以访问其资源，从而合法地绕过同源策略。

## 5. 文章服务实现 ArticleServiceImpl.java

```java
@Service
public class ArticleServiceImpl extends ServiceImpl<ArticleMapper, Article> 
        implements ArticleService {
    
    // 分页查询，支持关键词搜索、分类筛选
    @Override
    public PageResult<Article> getArticleList(Integer current, Integer size, 
                                               String keyword, Long categoryId, Integer status) {
        Page<Article> page = new Page<>(current, size);
        LambdaQueryWrapper<Article> wrapper = new LambdaQueryWrapper<>();
        
        // 关键词搜索标题和摘要
        if (StringUtils.hasText(keyword)) {
            wrapper.like(Article::getTitle, keyword)
                   .or()
                   .like(Article::getSummary, keyword);
        }
        
        if (categoryId != null) {
            wrapper.eq(Article::getCategoryId, categoryId);
        }
        
        if (status != null) {
            wrapper.eq(Article::getStatus, status);
        }
        
        // 置顶的排前面，然后按发布时间降序
        wrapper.orderByDesc(Article::getIsTop)
               .orderByDesc(Article::getPublishTime);
        
        IPage<Article> result = this.page(page, wrapper);
        return new PageResult<>(result.getTotal(), result.getRecords(), 
                                result.getCurrent(), result.getSize());
    }
    
    // 浏览量+1（每次查看文章详情时调用）
    @Override
    public boolean incrementViewCount(Long id) {
        Article article = this.getById(id);
        if (article != null) {
            article.setViewCount(article.getViewCount() + 1);
            return this.updateById(article);
        }
        return false;
    }
    
    // 获取热门文章（isHot=1，按浏览量排序）
    @Override
    public List<Article> getHotArticles(Integer limit) {
        LambdaQueryWrapper<Article> wrapper = new LambdaQueryWrapper<>();
        wrapper.eq(Article::getStatus, 1)
               .eq(Article::getIsHot, 1)
               .orderByDesc(Article::getViewCount)
               .last("LIMIT " + limit);  // 直接拼接LIMIT子句
        return this.list(wrapper);
    }
}
```

**为什么这样设计**：LambdaQueryWrapper使用方法引用而不是字符串指定字段名，避免字段名写错。分页查询自动计算SQL的LIMIT和OFFSET，同时查询总记录数。

**可能的问题**：如果并发用户同时点赞，浏览量会不会不准确？

回答：是的，这是一个并发问题。当前实现是先查询再更新，存在并发不安全的情况。更好的做法是使用SQL的原子操作：UPDATE article SET view_count = view_count + 1 WHERE id = ?。或者使用Redis原子计数器，定期同步到数据库。
