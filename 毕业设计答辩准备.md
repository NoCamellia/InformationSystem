# 信息咨询系统 - 毕业设计答辩准备材料

## 一、项目概述

### 1.1 项目背景
随着移动互联网的快速发展，用户对信息获取的需求日益增长。本项目旨在开发一个基于微信小程序的信息咨询平台，为用户提供便捷的资讯浏览服务，同时为管理员提供完善的内容管理系统。

### 1.2 项目目标
- 为用户提供便捷的资讯浏览体验
- 实现内容的分类管理和个性化推荐
- 提供完善的后台管理系统
- 支持用户互动功能（点赞、收藏、分享）

### 1.3 项目特色
1. **前后端分离架构**：采用现代化的开发模式，提高开发效率和可维护性
2. **微信小程序原生开发**：充分利用微信生态，降低用户使用门槛
3. **响应式管理后台**：美观的渐变色UI设计，良好的用户体验
4. **完整的权限管理**：基于RBAC的角色权限控制

- Composition API提供更好的逻辑复用

## 四、数据库设计

### 4.1 核心表结构

#### 1. 用户表 (user)
```sql
- id: 用户ID（主键）
- openid: 微信openid（唯一）
- nickname: 昵称
- avatar: 头像URL
- gender: 性别
- province/city/country: 地理信息
- status: 状态（正常/禁用）
- create_time/update_time: 时间戳
```

#### 2. 文章表 (article)
```sql
- id: 文章ID（主键）
- title: 标题
- summary: 摘要
- cover_image: 封面图
- content: 内容（富文本）
- category_id: 分类ID（外键）
- author_id: 作者ID（外键）
- view_count/like_count/collect_count: 统计数据
- is_top/is_hot/is_recommend: 标记
- status: 状态（草稿/已发布/已下架）
- publish_time: 发布时间
```

#### 3. 分类表 (category)
```sql
- id: 分类ID（主键）
- category_name: 分类名称
- parent_id: 父级ID（0表示顶级）
- level: 层级（1/2/3）
- sort_order: 排序
- status: 状态
```

### 4.2 表关系设计
- **一对多关系**：
  - 分类 → 文章
  - 管理员 → 文章
  - 角色 → 管理员
  
- **多对多关系**：
  - 文章 ↔ 标签（通过article_tag关联表）
  - 角色 ↔ 权限（通过role_permission关联表）
  - 用户 ↔ 文章（点赞、收藏关系）

### 4.3 索引设计
- **主键索引**：所有表的id字段
- **唯一索引**：openid、username、role_code等
- **普通索引**：category_id、author_id、status、publish_time等高频查询字段
- **全文索引**：article表的title和content字段（支持中文搜索）

## 五、关键技术实现

### 5.1 微信小程序登录流程

```javascript
// 1. 调用wx.login获取code
wx.login({
  success: (res) => {
    const code = res.code
    // 2. 将code发送到后端
    wx.request({
      url: 'http://localhost:8080/api/miniapp/user/login',
      method: 'POST',
      data: { code },
      success: (result) => {
        // 3. 后端返回token
        const token = result.data.data.token
        // 4. 保存token
        wx.setStorageSync('token', token)
      }
    })
  }
})
```

**后端处理流程：**
1. 接收小程序传来的code
2. 使用code调用微信API获取openid和session_key
3. 查询数据库，如果用户不存在则创建新用户
4. 生成JWT token返回给小程序

### 5.2 JWT身份认证

**Token生成：**
```java
String token = Jwts.builder()

    .setIssuedAt(new Date())

    .signWith(SignatureAlgorithm.HS512, secret)
    .compact();
```

**Token验证：**
```java
Claims claims = Jwts.parser()
```

## 六、可能的面试问题及答案

### 6.1 项目相关问题

**Q1: 为什么选择前后端分离架构？**

A: 前后端分离有以下优势：
1. **开发效率高**：前后端可以并行开发，互不影响
2. **职责清晰**：前端专注UI和交互，后端专注业务逻辑
3. **易于维护**：代码结构清晰，便于后期维护和扩展
4. **可复用性强**：同一套后端API可以服务多个前端（小程序、Web、App）
5. **性能优化**：前端可以做静态资源CDN加速，后端可以独立扩展

**Q2: 如何保证系统的安全性？**

A: 采取了多层安全措施：
1. **身份认证**：使用JWT token进行无状态认证
2. **权限控制**：基于RBAC的角色权限管理
3. **SQL注入防护**：使用MyBatis Plus的参数化查询
4. **XSS防护**：前端对用户输入进行转义
5. **密码加密**：使用BCrypt加密存储密码
6. **HTTPS传输**：生产环境使用HTTPS加密通信

**Q3: 如何处理高并发访问？**

A: 可以从以下几个方面优化：
1. **数据库优化**：
   - 添加索引提高查询效率
   - 使用连接池管理数据库连接
   - 读写分离（主从复制）
   
2. **缓存策略**：
   - Redis缓存热点数据
   - 浏览器缓存静态资源
   
3. **负载均衡**：
   - Nginx反向代理
   - 多实例部署
   
4. **异步处理**：
   - 消息队列处理非实时任务
   - 异步更新统计数据

**Q4: 项目中遇到的最大困难是什么？如何解决的？**

A: 最大的困难是MyBatis Plus与Spring Boot 3.2的兼容性问题。

**问题表现**：启动时报错 `Invalid value type for attribute 'factoryBeanObjectType'`

**解决过程**：
1. 首先查看错误堆栈，定位到是Spring容器初始化时的问题
2. 搜索相关issue，发现是版本兼容性问题
3. 尝试升级MyBatis Plus到3.5.6
4. 仍然报错，最终将Spring Boot降级到3.1.5
5. 问题解决，系统正常启动

**经验总结**：
- 选择技术栈时要注意版本兼容性
- 遇到问题要善于查看官方文档和GitHub issues
- 保持技术栈的稳定性，不要盲目追求最新版本

### 6.2 技术细节问题

**Q5: MyBatis Plus相比MyBatis有什么优势？**

A: 
1. **无需编写XML**：通过BaseMapper提供基础CRUD操作
2. **条件构造器**：LambdaQueryWrapper提供类型安全的查询
3. **分页插件**：内置分页功能，使用简单
4. **自动填充**：支持创建时间、更新时间自动填充
5. **逻辑删除**：支持逻辑删除，无需修改业务代码
6. **代码生成器**：可以快速生成Entity、Mapper、Service等代码

**Q6: Vue 3的Composition API相比Options API有什么优势？**

A:
1. **更好的逻辑复用**：通过组合函数（composables）实现逻辑复用
2. **更好的类型推导**：TypeScript支持更好
3. **更灵活的代码组织**：相关逻辑可以组织在一起
4. **更小的打包体积**：tree-shaking更友好
5. **更好的性能**：响应式系统重写，性能提升

**Q7: 如何实现文章的全文搜索？**

A: 使用MySQL的全文索引：

```sql
-- 创建全文索引
FULLTEXT KEY `ft_title_content` (`title`, `content`) 
WITH PARSER ngram

-- 查询
SELECT * FROM article 
WHERE MATCH(title, content) AGAINST('关键词' IN NATURAL LANGUAGE MODE)
```

优点：
- 支持中文分词（ngram parser）
- 查询速度快
- 无需额外组件

如果数据量大，可以考虑使用Elasticsearch。

**Q8: 小程序的性能优化做了哪些？**

A:
1. **分包加载**：将不常用的页面放到分包
2. **图片优化**：使用webp格式，懒加载
3. **请求优化**：合并请求，使用缓存
4. **代码优化**：避免频繁setData，使用节流防抖
5. **包体积优化**：删除无用代码和资源

### 6.3 设计模式问题

**Q9: 项目中使用了哪些设计模式？**

A:
1. **单例模式**：Spring Bean默认是单例
2. **工厂模式**：Spring的BeanFactory
3. **代理模式**：Spring AOP、MyBatis的Mapper接口
4. **模板方法模式**：MyBatis Plus的BaseMapper
5. **策略模式**：不同角色的权限策略
6. **观察者模式**：Vue的响应式系统

**Q10: RESTful API设计原则是什么？**

A:
1. **资源导向**：URL表示资源，不是动作
2. **HTTP方法**：
   - GET：查询
   - POST：创建
   - PUT：更新
   - DELETE：删除
3. **状态码**：正确使用HTTP状态码
4. **无状态**：每个请求包含完整信息
5. **统一接口**：返回格式统一

示例：
```
GET    /api/articles       # 获取文章列表
GET    /api/articles/1     # 获取ID为1的文章
POST   /api/articles       # 创建文章
PUT    /api/articles/1     # 更新ID为1的文章
DELETE /api/articles/1     # 删除ID为1的文章
```
